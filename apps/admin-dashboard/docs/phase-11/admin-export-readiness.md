# Phase 11 — Admin Export Readiness Notes

**Date:** 2026-06-20
**Author:** GHOST3030
**Scope:** Document safe export needs and restrictions for future implementation

## Purpose

Document what data export capabilities are needed for admin operations
and define safety restrictions, without implementing a broad export system
in Phase 11.

## Export Needs Identified

### 1. Student Data Export

| Data | Source | Sensitivity |
|------|--------|------------|
| Student list | `GET /admin/users` | Medium — PII (names, emails) |
| Progress records | `GET /admin/students/:id/progress` | Low — aggregate counts |
| Assessment results | `GET /admin/assessment-results` | Medium — individual scores |
| Placement results | `GET /admin/placement/results` | Low — CEFR level assignments |

### 2. Content Data Export

| Data | Source | Sensitivity |
|------|--------|------------|
| Course catalog | `GET /admin/courses` | Low |
| Question bank | `GET /curriculum/questions` | Medium — contains assessment content |
| Skill definitions | `GET /admin/skills` | Low |

### 3. Operational Data Export

| Data | Source | Sensitivity |
|------|--------|------------|
| Audit logs | `GET /admin/audit-logs` | Medium — user activity |
| Session summaries | `GET /admin/session-summaries` | Low — aggregate data |
| Report summaries | `GET /admin/reports/*` | Low — counts only |

## Safety Restrictions

Any future export implementation **must** follow these rules:

### Must Do

1. **Server-side generation** — Export files must be generated by the backend,
   not assembled client-side from paginated API calls
2. **Role-scoped access** — Only `pilot_admin` and `project_owner` roles
   may trigger exports
3. **Audit logging** — Every export action must be recorded in the audit log
   with the exporting user, timestamp, and data scope
4. **Rate limiting** — Exports must be rate-limited to prevent abuse
5. **PII handling** — Student PII exports must comply with data protection
   requirements (to be defined by the project owner)

### Must Not Do

1. **No client-side data assembly** — Never paginate through all records
   client-side to build an export file
2. **No raw database access** — Export endpoints must go through the same
   API layer as the admin UI
3. **No AI provider data** — Never export raw AI prompts, completions,
   or provider-specific data
4. **No AIM Engine internals** — Never export raw mastery vectors, weakness
   matrices, or recommendation model parameters
5. **No secrets in exports** — API keys, tokens, or credentials must never
   appear in exported data

## Recommended Export Formats

| Format | Use Case | Priority |
|--------|----------|----------|
| CSV | Tabular data (students, results, courses) | High |
| PDF | Formatted reports (assessment summaries) | Medium |
| JSON | API-compatible data interchange | Low |

## Implementation Approach (Future Phase)

1. Add backend export endpoints: `POST /admin/exports/:type`
2. Backend generates file asynchronously, returns download URL
3. Admin UI shows export status and download link
4. Files expire after configurable TTL (e.g., 24 hours)
5. All exports logged in audit trail

## Phase 11 Status

**No export system is implemented in Phase 11.** This document serves as
the readiness specification for future implementation. The admin reports
UI (P11-059) provides read-only display of backend summaries only.

## Conclusion

Export capabilities are a planned feature that requires backend endpoint
development, security review, and data protection compliance. Phase 11
documents the requirements and restrictions; implementation is deferred
to a future phase.
