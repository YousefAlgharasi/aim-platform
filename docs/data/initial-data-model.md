# AIM Initial Data Model and Entity List

## Purpose

This document drafts the initial logical data model for the AIM platform. It defines all core entities, their key fields, relationships, and the role each entity plays for users, AIM Engine, analytics, audit, and reporting. It is the authoritative planning reference for backend database design in Phase 1.

## Scope

Phase 0 planning documentation only. No database migrations, ORM code, SQL DDL, backend API code, Flutter code, or admin dashboard code is produced here. This document uses logical entity descriptions and field tables. Physical implementation decisions (indexes, partitioning, foreign key enforcement) belong to Phase 1.

## Entity Definitions

### 1. User

Represents every authenticated account in the platform regardless of role.

| Field | Type | Notes |
|---|---|---|
| `id` | UUID (PK) | System-generated. |
| `email` | string (unique) | Supabase Auth email. Never exposed to other students. |
| `role` | enum | One of: `student`, `pilot_admin`, `content_manager`, `human_reviewer`, `parent_guardian`, `project_owner`, `system_service`. |
| `display_name` | string | Safe display name. Not necessarily the learner's real name. |
| `auth_provider` | enum | One of: `email_password`, `google`, `magic_link`. |
| `is_active` | bool | False for suspended or deactivated accounts. |
| `created_at` | datetime (UTC) | |
| `updated_at` | datetime (UTC) | |

**Relationships:**
- One User may have one StudentProfile (if role = `student`).
- One User may be linked to one or more StudentProfiles via ParentGuardianLink (if role = `parent_guardian`).
- One User may appear in many AuditLog entries.

### 4. Skill

A unit of English learning from the skill tree. Canonical reference for AIM Engine and content.

| Field | Type | Notes |
|---|---|---|
| `id` | UUID (PK) | Matches skill ID in `docs/learning/english-skill-tree.md`. |
| `name` | string | e.g., "Present Perfect". |
| `category` | enum | One of: `PHO`, `VOC`, `GRA`, `READ`, `WRITE`, `LIS`, `SPE`. Canonical codes from `docs/learning/english-skill-tree.md`. |
| `level` | int (1–4) | Difficulty level within the category. |
| `parent_skill_id` | UUID (FK → Skill) or null | Hierarchical parent skill. Null for root-level skills. |
| `description` | text | Human-readable description for content managers and reviewers. |
| `is_active` | bool | Whether the skill is currently in use. Inactive skills are hidden from new lesson assignments. |

**Relationships:**
- Skills may have many prerequisite Skills (modeled in `SkillPrerequisite` join entity).
- Skills appear in Questions, LessonBlocks, StudentSkillStates, and AIMRecommendations.

```
 │    ├── PlacementTestResult (1:N)

 │    ├── ReviewSchedule (1:N)

Skill
 ├── SkillPrerequisite (M:M self-join)

 │    └── QuestionChoice (1:N)
 └── StudentSkillState (N:1)

AdminOverride → AuditLog (1:1 link)

```

## Assumptions

- Primary key type is UUID for all entities to support distributed inserts and privacy (no sequential IDs exposed externally).
- The platform uses Supabase PostgreSQL in Phase 1. All entities are PostgreSQL tables.
- `StudentSkillState` maps directly to the ORM model implemented in T-03 of the AIM algorithm repository. The entity here is the planning-level equivalent.
- `AuditLog` is always append-only. No UPDATE or DELETE operations are applied to audit records.
- `ProgressReport` is generated by the backend and cached as a snapshot. It is not computed live per request in MVP.

## Open Questions

| Question | Current Handling |
|---|---|
| Should `QuestionAttempt` store the full rendered question at time of attempt, or only the `question_version_id`? | Store `question_version_id`. Full question reconstruction from version is a content management concern. |
| Should `AIMRecommendation` store the full student state snapshot at recommendation time? | Store `mastery_before` for the primary skill. Full state snapshot is expensive. Full audit is available via AuditLog `before_state`. |
| Should `MicroGoal` be auto-generated on session end, or on a scheduled daily basis? | Open decision. AIM Engine generates on session end for simplicity in MVP. Scheduled generation is a post-MVP improvement. |
| Should `ProgressReport` be generated on demand or on a scheduled cadence? | Open decision. On demand (with caching) for MVP. Scheduled weekly generation for parent summary is a candidate feature. |
| Does `ParentGuardianLink` need admin approval, or can a parent self-link? | Defer to privacy planning (P0-022). Default: requires admin approval in MVP pilot. |
