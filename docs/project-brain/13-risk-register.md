| Risk | What it actually is | How found | Severity (as determinable) |
|---|---|---|---|
| ~~RLS disabled on 8 tables~~ | Was: 8 tables fully exposed to anon/authenticated Supabase roles, including `student_level_state` which drives real course-gating decisions. | Live Supabase advisor + `pg_tables` query | **Resolved** 2026-07-04 (migration applied, verified live) |
| Committed live credentials | `services/backend-api/backend.env` was tracked in git with a real service-role key, DB URL, JWT secret, provider keys. Untracked this session, but the keys remain in git history. | Direct file read | **High** until keys are rotated — untracking alone does not invalidate an already-exposed key |
| No CI coverage for `apps/web` | Zero workflow references `apps/web` in `.github/workflows/*`; it hosts the only Parent Dashboard in the repo. | Grep across every workflow file | **Medium** — a real, deployed-looking feature with no automated regression protection |
| `apps/web` duplicates `admin-dashboard` | Three feature areas (admin-ai, admin-analytics, admin-notifications) exist independently in both apps with separate API clients and separate test suites; unclear which is canonical. | Direct file-tree comparison across both apps | **Medium** — maintenance/drift risk, and a real question of which surface is actually live in production (Unknown) |
| AIM Engine is a hard, blocking dependency | `sessions.controller.ts:258` awaits `AimPipelineOrchestratorService.trigger(...)` synchronously in the request path; no feature flag exists to disable this call path. | Direct code read | **Medium-High** — if the engine is slow/down, every gated submission response is delayed; failure is caught (doesn't 500) but latency risk is real and un-mitigated |
| Payment provider not actually implemented | `PAYMENT_PROVIDER_ADAPTER` bound to `{}` in `billing.module.ts`; calling any adapter method throws. Yet billing tables have real row counts (`subscriptions`: 7, `checkout_sessions`: 9, `billing_entitlements`: 36). | Direct code read + live DB row counts | **High if billing is expected to be live** — either this data is fixture/test-only (benign) or real checkout flows are currently broken (severe); which one is **Unknown** |
| Notification delivery never actually dispatches | `NotificationDeliveryWorker.processQueue()` is never invoked in production code; push/email adapters are no-op stubs. `notification_events` has 62 real rows (enqueued), meaning real notifications are being silently dropped rather than delivered. | Grep for `processQueue` call sites (only in the worker + its unit test) + grep for real push/email SDK imports (zero found) | **High if users are expected to receive notifications today** — this is a live functional gap, not just a documentation gap |
| Migration ledger duplicates (pre-existing, not fully cleaned) | `_prisma_migrations` has several migrations recorded 2-3× (e.g. `add_assessment_db_constraints` ×3) predating this session. Only the 11 *missing* entries were reconciled this session; the duplicates were not investigated/cleaned. | Live SQL query grouping by `migration_name` | **Low-Medium** — didn't block anything found this session, but unverified against a real `prisma migrate deploy` run (couldn't reach `DATABASE_URL` from this sandbox) |
| `services/api` framing is stale across multiple docs | `PROJECT_STATE.md` and (until this session's update) `project-memory.json` call it "reference-only, zero live imports" — it is actually a live runtime dependency of `services/aim-engine`. | Direct code read + repo-wide grep, classified every hit as real-import vs. comment-only | **Low as a bug, Medium as an architecture-boundary risk** — anyone editing `services/api` believing it's dead reference code could silently change live AIM Engine behavior |
| Endpoint prefix inconsistencies | Three confirmed real inconsistencies: `notifications`/`parents` controllers hardcode `api/v1/...` while everything else is bare; `analytics/exports` omits the `/admin` prefix its siblings use; placement-test-status uses a reversed `/placement/admin/...` prefix. | Direct grep of every `@Controller()` decorator | **Medium** — concrete 404 risk for student-web depending on deployed `REACT_APP_API_BASE_URL` (Unknown), and a real developer-confusion risk regardless |
| 5 orphaned backend services | `AiTeacherFallbackService`, `AnswerSubmissionService`, `QuestionDeliveryService` (assessments), `ParentAccessAuditService`, `AudioUploadSafeFailureService` — built, never imported by any module. | Exhaustive grep of every class name against every `*.module.ts` | **Low** — dead code risk (maintenance burden, confusion for future readers) rather than a functional bug, since nothing calls them |
| No feature flag around Phase 20/21 rollout | Confirmed absent in code; every student's next lesson/assessment/placement attempt calls the real AIM pipeline with no staged/cohort rollout mechanism. | Direct code read, cross-checked against `PROJECT_STATE.md`'s own risk note | **Medium** — no ability to dial back the Phase 20/21 rollout if a problem emerges in production |
