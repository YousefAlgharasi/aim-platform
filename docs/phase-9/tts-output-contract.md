# Phase 9 — Text-to-Speech Output Contract

## Purpose

This document defines the exact shape of the output the TTS Gateway
(Group D) returns to Voice Session Orchestration after synthesizing the
filtered AI Teacher reply text into audio, per the pipeline defined in
`docs/phase-9/voice-architecture.md` and
`docs/phase-9/voice-request-lifecycle.md` (step 10). It keeps TTS
output predictable and provider-agnostic, so no other backend component
needs to understand a specific TTS provider's response format, and so
no TTS provider detail or raw audio location ever leaks past the
Gateway.

---

## Output Contract

A successful TTS Gateway call returns this shape internally:

```jsonc
{
  "audioRef": "string — opaque reference to backend-managed storage",
  "durationMs": "number — synthesized audio duration in milliseconds",
  "contentType": "string — MIME type of the synthesized audio, e.g. \"audio/mpeg\""
}
```

- **`audioRef`** is generated by the Gateway (or by Voice Persistence
  immediately after storing the synthesized audio) and is the only
  field used to retrieve the audio later, via
  `GET /ai-teacher/voice/audio/:audioRef` (per
  `docs/phase-9/voice-api-map.md` §3). It is never a provider URL, a
  filesystem path, or raw audio bytes returned inline to the caller of
  the Gateway.
- **`durationMs`** is an internal observability value (e.g. for
  logging or client playback UI hints); it is not a learning-decision
  field and is not required to be exposed to the client beyond what the
  API map already declares.
- **`contentType`** lets the Voice API Endpoint set the correct
  `Content-Type` header when streaming the audio back to Flutter via
  endpoint 3, without the endpoint needing to know which TTS provider
  produced the audio.

No other top-level field is part of the contract. In particular, the
following are never included in TTS Gateway output:

| Excluded field | Reason |
|---|---|
| `provider`, `model`, `voiceId`, `rawProviderResponse` | Provider-specific detail stays inside the TTS Gateway; no other component needs to know which provider or voice produced the audio (`docs/phase-9/no-client-provider-rule.md`). |
| `masteryScore`, `level`, `weakness`, `difficulty`, `recommendation`, `reviewSchedule`, or any similarly named field | The TTS Gateway only synthesizes audio from already-filtered text; it never computes or attaches a learning-decision value (`docs/phase-9/no-aim-authority-change-rule.md`). |
| `apiKey`, `endpoint`, or any provider credential/config value | Provider configuration is read once inside the Gateway from environment/config; it never appears in the Gateway's output object. |
| Raw audio bytes inline in this contract object, a provider audio URL, or a filesystem path | Audio bytes are written to backend-managed storage and referenced only via `audioRef`; this contract never embeds the audio payload itself or a provider/file location (`docs/phase-9/voice-scope-boundaries.md` — "Excluded Scope: Committing generated private audio files"). |

---

## Failure Contract

If the TTS provider call fails, times out, or returns an unusable
result, the TTS Gateway does not return a partial or placeholder
`audioRef`. It raises a backend-internal error that the calling
component (Voice Session Orchestration) maps to the safe error response
described in `docs/phase-9/voice-request-lifecycle.md` (failure point
"TTS Gateway call fails/times out, step 10") and
`docs/phase-9/voice-api-map.md` (endpoint 1 error list). No raw provider
error text, stack trace, or provider error code crosses this boundary.
The already-filtered reply text (from step 9) may still be returned to
the client without audio, or the whole turn may be marked failed,
exactly as decided by the implementing orchestration task's error
contract — this document only fixes the Gateway's own output/failure
shape, not that downstream policy choice.

---

## Predictability Rules

- `audioRef` is always a non-null string when the call succeeds; the
  Gateway never returns an empty string or a placeholder value in place
  of a real reference.
- `audioRef` is opaque to every caller outside backend-managed storage
  resolution — it carries no embedded provider name, file extension
  implying a raw path, or predictable/guessable pattern that would let
  a client infer another student's reference.
- `contentType` is always one of a small, backend-declared set of audio
  MIME types; it is not a free-form string copied verbatim from the
  provider response.

---

## Where This Contract Applies

- **TTS Gateway (Group D):** the sole producer of this shape; the sole
  component that talks to the configured TTS provider.
- **Voice Session Orchestration (Group B/E):** the sole consumer of
  this shape; treats a Gateway failure per the Failure Contract above.
- **Voice Persistence (Group E):** persists only `audioRef` (as part of
  the Voice Turn shape defined in
  `docs/phase-9/voice-session-contract.md`), never `durationMs`,
  `contentType`, or any provider-specific field from this contract.
- **Voice API Endpoints (Group F):** the audio-stream endpoint (§3 of
  `docs/phase-9/voice-api-map.md`) resolves `audioRef` to a byte stream
  using `contentType` for the response header; it never re-exposes this
  internal contract's shape directly to Flutter.

---

## Validation

- TTS output is defined as a predictable, provider-agnostic shape
  before implementation begins.
- The contract excludes any learning-decision field, provider
  credential, raw audio payload, or provider/file location.
- AI Teacher Voice Mode does not replace AIM Engine authority via this
  contract.
- Flutter does not call a TTS provider directly; only the TTS Gateway
  talks to the provider.
- No secrets or generated private audio files are referenced or
  committed in this document.
