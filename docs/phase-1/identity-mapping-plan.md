# Identity Mapping Plan

## Purpose

This document defines how Supabase Auth identities map to AIM Platform backend identity records during Phase 1 and later implementation.

The goal is to prevent identity mismatch between:

- Supabase Auth users.
- Backend `User` records.
- Learner/student profile records.
- Parent or guardian relationship records.
- Role and authorization checks.
- Ownership checks across learner, parent, admin, and internal surfaces.

This plan is documentation-only. It does not create database schema, migrations, runtime code, Prisma models, Supabase policies, NestJS guards, Flutter models, or Admin Dashboard code.

## Source of Truth

Supabase Auth is the source of truth for authentication identity.

The backend database is the source of truth for application profile data, platform roles, ownership relationships, and authorization metadata.

## Identity Anchor

The Supabase Auth user ID must be treated as the stable external authentication identifier.

Recommended naming:

| Concept | Recommended Field | Notes |
|---|---|---|
| Supabase Auth UID | `auth_user_id` | Stores Supabase Auth `auth.users.id`. |
| Backend User ID | `id` | Internal backend user record primary key. |
| Learner Profile ID | `id` | Internal learner/student profile record primary key. |
| Parent Profile ID | `id` | Optional future parent profile record primary key. |

## Core Rule

A backend user record must map to exactly one Supabase Auth UID.

A Supabase Auth UID must map to exactly one backend user record.

This prevents one person from having multiple conflicting backend identities unless an explicit future account-linking policy is approved.

## Recommended User Mapping

Future schema should follow this conceptual relationship:

```text
Supabase Auth User
  auth.users.id
        |
        | maps to
        v
Backend User
  users.auth_user_id
        |
        | owns profile(s)
        v
Learner / Parent / Admin profile records
```

Recommended conceptual table fields:

```text
users
- id
- auth_user_id
- email
- display_name
- role
- status
- created_at
- updated_at
```

Rules:

- `users.auth_user_id` must be unique.
- `users.auth_user_id` must not be nullable after user creation.
- `users.auth_user_id` must not be reused across users.
- Auth UID must not be generated by the backend.
- Auth UID must come from a verified Supabase Auth token.
- The backend must not trust a client-provided `auth_user_id`.

## StudentProfile Relationship

Learner/student profile data should be separate from base identity.

Recommended conceptual relationship:

```text
users.id
  |
  | one-to-one or one-to-many depending on later product decision
  v
student_profiles.user_id
```

Recommended Phase 1 default:

- One backend user may have one learner profile.
- Parent-child relationships should be represented separately if parent access is later enabled.
- Student profile records must not be used as authentication identity.

Recommended conceptual table fields:

```text
student_profiles
- id
- user_id
- display_name
- preferred_language
- timezone
- learning_goal
- created_at
- updated_at
```

Rules:

- `student_profiles.user_id` must reference `users.id`.
- A learner profile must not exist without a backend user.
- A learner profile must not authenticate directly.
- A learner profile must not contain Supabase provider credentials.
- A learner profile must not be used as the source of role truth.

## Parent Relationship Mapping

Parent access is optional and later-phase scoped.

If enabled, it should use explicit relationship records.

Recommended conceptual relationship:

```text
parent_user.id
  |
  | relationship
  v
student_user.id
```

Recommended conceptual table fields:

```text
user_relationships
- id
- parent_user_id
- child_user_id
- relationship_type
- status
- created_at
- updated_at
```

Rules:

- Parent access must be authorized through explicit relationship records.
- A parent must not access a learner by guessing `user_id`, `student_profile_id`, email, phone number, or display name.
- Parent access must not expose internal AIM fields.
- Parent access must be scoped by backend ownership checks.

## Role Source of Truth

The backend database is the source of truth for AIM platform roles.

Supabase Auth authenticates the user; the backend determines what the user may do.

Recommended role values should align with `packages/shared-contracts/enums/common-enums.md`.

Suggested role values:

- `LEARNER`
- `PARENT`
- `INSTRUCTOR`
- `CONTENT_MANAGER`
- `REVIEWER`
- `ADMIN`
- `SUPER_ADMIN`

Rules:

- Do not trust role values sent by clients.
- Do not use Flutter-local role values for authorization.
- Do not use client-side decoded JWT claims as final authorization truth.
- Backend must resolve role after verifying the Supabase Auth token.
- Admin role elevation must require backend-controlled update paths.
- Role changes must be auditable in later implementation.

## Auth Request Flow

Recommended future request flow:

```text
Client
  |
  | sends Supabase access token
  v
Backend API
  |
  | verifies token
  v
Resolved Auth Identity
  |
  | finds users.auth_user_id
  v
Backend User
  |
  | resolves role + ownership
  v
Authorized handler
```

Rules:

- Client sends token only.
- Client does not send trusted role.
- Client does not send trusted ownership.
- Client does not send trusted `auth_user_id`.
- Backend resolves all identity and authorization context.

## Ownership Check Implications

Ownership checks must use backend-resolved identity.

Do not perform ownership checks using raw client input.

Allowed ownership anchors:

| Resource Type | Ownership Anchor |
|---|---|
| User profile | `users.id` resolved from verified `auth_user_id`. |
| Student profile | `student_profiles.user_id`. |
| Parent access | `user_relationships.parent_user_id` and `child_user_id`. |
| Session records | Backend-resolved learner `users.id` or profile owner. |
| Submissions | Backend-resolved learner owner. |
| Progress records | Backend-resolved learner owner. |
| Notifications | Backend-resolved user owner. |

Forbidden ownership methods:

- Client-provided `user_id`.
- Client-provided `student_profile_id`.
- Client-provided role.
- Client-provided parent-child relationship.
- Email matching from request body.
- Display name matching.
- Local Flutter state.
- Query string identity parameters unless backend verifies them against resolved identity.

## Identity Mismatch Risks

### Risk: Duplicate Backend Users for One Supabase UID

Impact:

- Split progress history.
- Incorrect recommendations.
- Wrong ownership checks.
- Broken parent relationships.

Mitigation:

- Enforce unique `users.auth_user_id`.
- Use idempotent user provisioning.
- Reject conflicting identity creation.

### Risk: One Backend User Linked to Wrong Supabase UID

Impact:

- Account takeover risk.
- Cross-user data leak.
- Wrong learner progress exposure.

Mitigation:

- Never accept `auth_user_id` from client body.
- Resolve UID only from verified token.
- Audit identity-link changes.
- Restrict manual identity repair to admin-only controlled workflow.

### Risk: StudentProfile Used as Auth Identity

Impact:

- Bypasses base user ownership rules.
- Breaks parent/admin authorization.
- Creates inconsistent role checks.

Mitigation:

- Treat StudentProfile as profile data only.
- Always resolve through `users.id`.
- Keep role on User or a backend-controlled role relation, not profile.

### Risk: Role Drift Between Supabase and Backend

Impact:

- Client may show wrong UI.
- User may appear to have unauthorized privileges.
- Admin operations could become unsafe.

Mitigation:

- Backend role is final source of truth.
- Client role is display-only.
- Backend validates every privileged operation.

### Risk: Parent Access Without Relationship Verification

Impact:

- Parent may see unrelated learner data.
- Learner privacy breach.

Mitigation:

- Require explicit `user_relationships` record.
- Require active relationship status.
- Use backend ownership guard for every parent-scoped endpoint.

## Provisioning Strategy

Recommended future provisioning flow:

1. User signs in or signs up through Supabase Auth.
2. Backend receives a request with a verified Supabase access token.
3. Backend resolves `auth_user_id`.
4. Backend looks up `users.auth_user_id`.
5. If no user exists, backend creates one through a controlled provisioning path.
6. Backend creates only the minimum profile data required.
7. Backend returns learner-safe profile fields only.

Rules:

- Provisioning must be idempotent.
- Provisioning must not create duplicate users for the same `auth_user_id`.
- Provisioning must not assign admin roles by default.
- Provisioning must not trust client-provided role.
- Provisioning must not create parent-child relationships automatically without explicit workflow.

## Data Exposure

Backend responses must follow `docs/phase-1/safe-field-exposure-contract.md`.

Identity-related fields exposed to clients should be minimized.

Learner-safe examples:

```text
user.id
user.display_name
user.role
profile.preferred_language
profile.timezone
```

Internal-only examples:

```text
users.auth_user_id
raw JWT claims
provider subject
role resolution trace
ownership check trace
```

## Supabase RLS Implications

Supabase RLS planning must account for the backend identity model.

Phase 1 guidance:

- RLS policies must not rely on client-provided identity.
- RLS policies must be planned separately from Prisma schema.
- Backend service access must not leak service-role keys.
- Direct client database access is not part of Phase 1 learner flow.
- Flutter Mobile must use Backend API, not direct database writes for protected learning records.

## AIM Engine Implications

AIM Engine must not own user identity.

AIM Engine may receive backend-approved internal identifiers for processing, but:

- It must not authenticate users.
- It must not authorize user access.
- It must not expose records directly to clients.
- It must not become source of role truth.
- Backend API must mediate AIM outputs.

## AI Teacher Implications

AI Teacher gateway must use backend-resolved identity context.

Rules:

- Client must not pass trusted user role to AI Teacher.
- Client must not pass trusted learner ownership to AI Teacher.
- Backend must attach safe identity context.
- AI Teacher must not receive secrets or raw authorization internals.
- AI Teacher final output must be mediated by Backend API.
